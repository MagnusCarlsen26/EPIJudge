# Content Hashing For Frontend CDN Data

CDN-ready data is generated by running:

```bash
cd frontend
npm run build:data
```

The Vercel `build` script only writes `config.js`; it does not regenerate data. This keeps deploys working even after the original judge source and solution folders are removed from the deploy branch.

The data generator lives in `frontend/scripts/build-data.mjs`.

Hashing uses Node's built-in `crypto` module:

```js
createHash("sha256").update(value).digest("hex").slice(0, 12)
```

That means each generated filename uses the first 12 hex characters of the SHA-256 hash of its content.

## Boilerplate Files

Each starter file is read from the language-specific judge folder:

```text
epi_judge_python/<problem>.py
epi_judge_cpp/<problem>.cc
epi_judge_java/<Problem>.java
```

The file content is hashed directly. The emitted file path is:

```text
frontend/public/data/boilerplate/<language>/<problem>.<hash>.<ext>
```

Example:

```text
two_sum.1e9220d5642f.py
```

If the starter code changes, the hash changes, so the filename changes.

## Solution Files

Solution files are generated with the same hash strategy from:

```text
epi_judge_python_solutions/
epi_judge_cpp_solutions/
epi_judge_java_solutions/
```

They are emitted as:

```text
frontend/public/data/solutions/<language>/<problem>.<hash>.<ext>
```

## Problems JSON

The build script creates one problem metadata JSON payload:

```text
{ "chapters": [...] }
```

It hashes the full formatted JSON string and writes:

```text
frontend/public/data/problems.<hash>.json
```

If any problem metadata or boilerplate path inside that JSON changes, the JSON hash changes.

## Manifest

The stable entrypoint is:

```text
frontend/public/data/manifest.json
```

It points to the current hashed files:

```json
{
  "problems": "/data/problems.<hash>.json",
  "assets": {
    "boilerplate": {
      "python": {
        "two_sum": "/data/boilerplate/python/two_sum.<hash>.py"
      },
      "cpp": {
        "two_sum": "/data/boilerplate/cpp/two_sum.<hash>.cc"
      },
      "java": {
        "two_sum": "/data/boilerplate/java/two_sum.<hash>.java"
      }
    },
    "solutions": {
      "python": {
        "two_sum": "/data/solutions/python/two_sum.<hash>.py"
      }
    }
  },
  "boilerplate": {
    "python": {
      "two_sum": "/data/boilerplate/python/two_sum.<hash>.py"
    }
  }
}
```

Vercel should revalidate `manifest.json`, while hashed files can be cached immutably.

The manifest includes both problem-id maps and filename maps:

```text
assets.boilerplate.python[problemId]
assets.solutions.python[problemId]
assetsByFilename.boilerplate.java["CountBits.java"]
assetsByFilename.solutions.cpp["count_bits.cc"]
```

The filename maps avoid losing entries when multiple mapped problems share the same basename.
